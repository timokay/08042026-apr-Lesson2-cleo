# Solution Strategy: CSV Upload

## Ключевые технические решения

### 1. Encoding detection без библиотек

**Решение:** Попытка decode UTF-8, при ошибке → cp1251.

**Обоснование:** `chardet` добавляет ~10 МБ зависимости и иногда ошибается на коротких файлах. Для российских банков практически 100% вариантов — это UTF-8 (веб-экспорт) или cp1251 (Windows-приложение). Двуступенчатая проверка надёжнее.

**Риск:** Экзотические кодировки (KOI8-R, IBM866) не поддерживаются. Принято — аудитория использует только Т-Банк веб.

### 2. Rule-based категоризация вместо AI

**Решение:** Regex-паттерны по названию мерчанта, AI только при низком confidence.

**Обоснование:** 
- Latency: regex < 1мс vs AI 1-3с
- Cost: 0 vs ~$0.002/1000 токенов
- Точность: 85% мерчантов покрываются паттернами
- AI batching (20 транзакций за вызов) как опциональное улучшение

### 3. Ограничение 1000 строк без hard-fail

**Решение:** Тихое усечение + banner "Показаны первые 1000 транзакций".

**Обоснование:** Полный fail при 2000+ строках = потерянный пользователь. 1000 транзакций достаточно для паттернов (обычно 3-6 месяцев). Пользователь видит предупреждение и может разбить файл.

### 4. BFF-прослойка Next.js

**Решение:** Браузер → Next.js /api/upload → AI service /analyze.

**Обоснование:** 
- AI service в docker-сети, не публичен
- user_id берётся из auth middleware (не из тела запроса браузера)
- Суpabase credentials только на Next.js server

### 5. Дедупликация по (date, amount, description)

**Решение:** Set-based dedup по составному ключу.

**Обоснование:** Т-Банк иногда дублирует строки при повторном экспорте. Без дедупликации пользователь видит удвоенные суммы.
